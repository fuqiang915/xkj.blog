Android 分辨率

名词解释：

Density : 值表示每英寸有多少个显示点，与分辨率是两个概念。 基于屏幕密度的抽象单位，设备无关的点，用于说明与密度无关的尺寸和位置。这些单位是相对于一个160dpi的屏幕，所有一个dp是160dpi屏幕上的一个点。

dp, sp, px的区别

  • dp与密度无关的像素, 160dip(每英寸屏幕160像素点)屏幕上1dp=1px
  • px实际像素值,屏幕上的点，大小永远不会变
  • spscaled pixels — best for text size (放大像素)（与刻度无关的像素）：主要处理字体的大小。与dp类似，但是可以根据用户的字体大小首选项进行缩放。
    主要用于字体显示best for textsize。虽然谷歌建议用sp作字体单位，但是实际应用的时候，用户若设置字体大小会破坏布局，增加布局难度，最好是用dp作字体单位。

  • layout 放一些通用布局xml文件，比如界面中顶部和底部的布局，不会随着屏幕大小变化，类似windos窗口的title bar
  • layout-small 屏幕尺寸小于3英寸左右的布局
  • layout-normal 屏幕尺寸小于4.5英寸左右，
  • layout-large 4英寸-7英寸之间
  • layout-xlarge 7-10英寸之间

Android Permissions

Android权限相关解析

安全架构

Android权限体系的关键点是，在默认情况下App没有任何权限去做一些可能对系统、用户或其他app不利的事,包括读写用户隐私数据，访问其他程序数据，网络接口，保持设备常亮等等。

因为每个App运行在独立沙盒中，必须明确声明需要共享的数据。可以通过声明自定义permissions。App静态声明需要的权限，系统会在安装app时获得用户的允许。

Application Signing

Android的签名作用在于

  • 用来区分每个app
  • 自签名，不需要权威机构授予签名，自己通过工具生成即可
  • 系统用来实现签名级别的权限(signature-level permissions)

Android Service详解

Service 四大组件之一，主要的后台任务组件，重点用于后台任务，比如播放音乐，做复杂运算，后台下载等等。
但是Service也是运行在主线程的，所以注意耗时操作。

实现一个Service的方式有以下几种

  • 继承Service
  • 继承IntentService

IntentService

官方实现 用来实现异步线程队列的Service，方便使用。
适用于后台线程异步任务，按顺序执行，执行完成自动关闭的功能。比如后台下载任务，上传图片任务都比较适合。

面向对象编程

Object Oriented Programing

六大原则：

  • 单一职责: 一个类只有一个改变的动机。一个类只有一个职责。
  • 开闭原则: 对修改关闭，对扩展开放
  • 里氏替换原则: 父类可以由子类替换，而不出现错误
  • 依赖倒置原则: 依赖抽象而不是实现.面向抽象编程，而不是实现
  • 接口隔离原则: 一个类对另一个类的依赖应该建立在最小接口上
  • 迪米特原则: 一个对象应该尽可能独立，对其他类了解越少越好

Monkey日志分析

本文讲解Monkey日志的分析方法，帮助解决常见的Monkey日志中的bug出现位置的查找方法

Monkey

Monkey是Android客户端用于做压力测试的工具，Google官方提供的开源工具。运行Monkey后会根据参数发送随机事件到客户端模拟用户操作来进行压力测试，当出现崩溃时会自动保存相关日志，并发送邮件给开发者。

一个Monkey崩溃日志一般包括三个部分:

  • xxxxxx_x_xxxxxx_bugreport

    这个日志基本包含需要的大部分需要的信息，是重点分析对象，里面包含Android系统的相关信息：

    • 运行中的Activity : Running Application Activities
    • 运行中的Service : Running Services
    • 运行中的Provider : Running Application Providers
    • 系统Log : SYSTEM LOG
    • 事件日志 : EVENT LOG
    • CPU信息: CPU INFO (top -n 1 -d 1 -m 30 -t)
    • 虚拟机内存信息： VIRTUAL MEMORY STATS
    • 虚拟机内存分配情况：VMALLOC INFO
    • 线程信息等等

重点日志一般在系统Log和Activity,里面包含了崩溃信息和Activity的Task信息,大部分错误都可以通过这两种日志找到,至于其他的就按照其他日志按图索骥吧。

搜索 “—– ” 可以定位到每种日志的开头,或者直接搜索上面列出的名称

  • xxxxxx_x_xxxxxx_Monkey_Error

    直接的Android错误堆栈信息,里面一般可以看到出错类的位置，这个可以直接找出错误位置，但是里面指出的错误一般只有一个，但是崩溃可能不止这一个，更多的还是需要在bugreport中寻找。具体信息如下：

  • xxxxxx_x_xxxxxx_Monkey_Standard

    里面是Monkey发送事件的日志，可以看到Monkey 总共事件数和Seed以及所有Monkey发送的随机事件，可以用来了解Monkey所有的测试步骤，对于我们找问题起到的作用不大，如下：
