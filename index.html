Android 自定义通知一些坑

本篇介绍自定义通知样式时会遇到的一些坑,本篇旨在记录坑不在全面讲解自定义的方法

坑

  • 文字颜色:强烈使用系统定义的文字颜色,否则需要固定背景色
  • 自定义View的尺寸: 建议自适应，因为不同系统高度会有差别
  • 不同系统版本的兼容性处理

详细讲解

文字颜色

首先一个大前提，通知栏的背景色的颜色是不确定的，有深色，也有浅色。
这里分两种情况说明。
一种是自定义布局的背景色固定，这种情况下可以直接指定文字颜色。但是这种在通知栏的观感不是很好，比如浅色通知栏，显示一条黑色通知显得很突兀，也不好看。

第二种是透明背景色，这时文字颜色就不能固定了，好在系统提供的对应的文字样式。

由于5.0引入MeterialDesign,5.0及以上的样式命名有所更改,所以需要分别设置。

自定义View的尺寸

首先Android原生系统中定义的样式大致有两种，一是普通通知，另一种是4.1以后引入的扩展通知
普通通知高度为64dp,扩展通知高度为256dp。但是由于国内不同定制系统对这两种高度有不同处理，所以推荐使用自适应的布局，尽量不要固定宽高。
至于通知的宽度，不只受屏幕宽度影响，也受自制系统影响。比如华为系统的通知左侧会有时间轴、小米和魅族系统左右会预留padding空间等等。

所以尺寸上尽量做到自适应。

不同系统版本的兼容

扩展布局只支持4.1以上系统，对于4.1以下，可以只设置普通视图的布局。
对于7.0以上系统一定不要直接使用notification.bigContentView来设置布局，此方法早已废弃，尤其7.0以后bigContentView可能为空值

一些代码细节

  • 创建通知可以使用v4包中的NotificatoinCompat.Builder,以兼容11以下版本。
  • 设置自定义布局推荐同时设置普通状态和扩展状态的布局，否则有些手机会出现 Bad Notification Exception

Java内存原理

本文详细讲解Java内存原理，旨在深入了解技术细节

首先上图，Java内存模型图:

下面我们挨个解释各个组成部分:

程序计数器

程序计数器是当前线程执行的字节码地址指示器。每个线程有自己的计数器，互相独立。字节码解释器就是通过改变程序计数器来指定下一步应该执行的指令。(虽然名字叫计数器，但是称为指令指示器好像更合适)

当执行的是Native方法时，计数器值为空。

这个区域是Java内存中唯一一个没有OutOfMemoryError的区域。就是说无论执行什么代码，需要的内存都是在范围内的，而其他区域就没这么简单了。

Java虚拟机栈

Java虚拟机栈保存的是方法执行过程中的数据，是线程私有的，生命周期与线程相同。栈里面操作单位是栈帧，栈帧中包含：方法的局部变量表、操作数栈、动态链表、方法出口信息等，一个方法执行需要的所有信息。每一个栈帧的入栈出栈对应了一个方法的开始执行到结束。

局部变量表中存放了编译器可知的各种基本数据类型(boolean、byte、char、short、int、float、long、double)、对象引用和returnAddress类型(指向了一条字节码指令的地址)。

每次执行一个方法会创建一个栈帧，当栈满时会抛出OutOfMemoryError。

本地方法栈

与Java虚拟机栈类似，是用来执行Native方法的方法栈。有的虚拟机将二者合二为一

会抛出stackOverflowError和OutOfMemoryError异常。

Java堆

Java堆是所有线程共享的一块内存区域，虚拟机启动时创建，此区域用来保存对象实例。

Java堆是GC的主要区域，可以细分为新生代和老年代。堆内存在物理上可以是不相连的，逻辑上相连即可。
可以固定大小，也可以实现为可扩展。

当Java堆没有足够内存，而且无法扩展时会抛出OutOfMemoryError

修改第三方jar包中class文件的方法

今天项目中有个第三方jar包有bug，需要修改，因为没有提供新版本，所以打算直接修改jar包。

首先是发现了jar包中一个类e中会因为某种原因抛出异常导致线程崩溃，会反馈到我们的CrashReport系统上。那么最直接的方法就是修改这个类。

我们知道class只是java编译后的二进制文件,我们只需要直接替换这个文件，理论上就可以解决问题。

那么如何修改？

首先直接把class类反编译后再利用javac重新编译这种方案是行不通的。因为有很多链接库找不到，编译无法通过。

网上找了一下现存的方案，有一个windows下的工具 jclasslib。大概看了一下，是把class文件中的常量池，方法列表等反编译出来，达到可编辑的状态。基本上可以用来修改常量池中的内容，比如一些字符串，类名，方法名等等，做小幅度修改是够了，但是并不能大量修改逻辑。

还是看一下怎么替换class文件吧。

直接替换class文件

上面说了替换文件是可以的实现修改的，关键问题在于怎么生成一个新的class—–聪明的孩纸也想到了:原来怎么生成，还怎么生成呗。
建一个Android项目重新编译！因为这里探讨的是Android平台，其他平台类似。

下面看看详细步骤：

  • 反编译旧class文件,生成java类代码文件
    反编译直接用AndroidStudio或者网上找反编译工具也行
  • 新建一个Android项目A，A中建一个moudle B,将反编译的java类文件+jar包导入moudle B 中
    这里就是为了重新将修改后的java类编译，这里注意需要将jar包中需要修改的class文件从里面删掉，否则类会重复，编译不过。
    至于删除方法
    • windows直接zip工具将jar解压缩，删除文件,再重新压缩为jar
    • mac系统下直接命令行 zip -d filePath就可以了
  • 修改java类逻辑，编译运行项目A
    此时java类需要保证类名，方法名什么的不变，不然jar中有其他依赖也没法编译不是~;内部逻辑就可以随便改了。
  • 把A项目编译后的class.jar中对应的新class文件导出
    jar包就在 module B 的目录下:build/intermediates/bundles/default/class.jar
    将其解压，导出对应类的class就可以拿去替换啦
  • 将新的class文件替换到第三方jar包中,搞定。

Android SQLite数据库升级处理

按照业务需求，需要做Android数据库升级，记录一下通常的实现方法

我们都知道，实现SqLite升级需要重写SQliteOpenHelper的onUpgrade方法,此方法会传递三个参数 db,oldVersion,newVersion

升级时需要根据当前版本和新版本来确定如何升级,这就需要处理顺序升级和跨版本升级。

实现方案

目前有两个方案供我们选择：

  • 方案1: 可以遍历所有升级情况，为每种情况实现一种升级方式
    例如，假设我们目前最新版本为4那么按照升级历史来说我们应该处理了下面几种情况：

    • 1->2
    • 2->3
    • 1->3
    • 1->4
    • 2->4
    • 3->4

  • 方案2: 按照版本顺序升级
    还是上面的例子，升级到版本4,我们需要处理三种情况:

    • 1->2
    • 2->3
    • 3->4

这里可以看到，方案1随着版本增加，需要考虑的情况会呈指数级增长，工作量也随之上升;但是它的优势在于每次升级版本都是直接从旧版本升级到最新版本,只做一次升级操作,效率是最高的,（相对于第二种方案）不会出现冗余的操作。

方案2的优势在于它考虑的情况少了很多，只需要关心两个版本间的差别，每次升级的工作量只是当前版本的修改，不会显著增加。但是缺点是需要挨个版本升级，假设一个极端情况从版本1，升级到版本10，会做10次升级操作，这其中如果冗余操作比较多的话可能会消耗较长时间，所以最好提供进度条来处理这种情况。

Java 基础细节问题总结

数据类型

整数

  • byte : 8位,1字节 -128~127
  • short: 2字节
  • int : 4字节
  • long : 8字节

浮点数

  • float : 4字节
  • double: 8字节

字符型

  • char : 2字节,可以表示一个中文字符

布尔

  • boolean : 1位,1/8字节

关键字

一些不太常用的关键字:

  • strictfp: strict float point 精确浮点数,可应用于类，接口，方法；此关键字修饰的类或方法，里面的float,double需要符合IEEE-754规范
  • transient: 标记不会被序列化的字段
  • volatile: 通常，volatile关键字用来阻止（伪）编译器认为的无法“被代码本身”改变的代码（变量/对象）进行优化。
  • assert: 用于程序调试

Volatile的官方定义

Java语言规范第三版中对volatile的定义如下： java编程语言允许线程访问共享变量，为了确保共享变量能被准确和一致的更新，线程应该确保通过排他锁单独获得这个变量。Java语言提供了volatile，在某些情况下比锁更加方便。如果一个字段被声明成volatile，java线程内存模型确保所有线程看到这个变量的值是一致的。

解释一下：

volatile实现了synchronized的可见性部分的功能，但是并不能保证互斥。也就是可以保证一个线程修改了共享变量的值，其他线程能马上读取到这个值；所以可以用于多线程读，而不能用于多线程写实现同步。
volatile的读的额外开销很小，而写的开销比较大，但是都比synchronized效率高不少。用volatile的主要原因就是在某些情况下可以提高性能。

Math

  • Math.floor(x); (向下取整)取小于等于x的最接近的double类型整数;参数是double
  • Math.round(x); 四舍五入,返回long或int;
  • Math.ceil(x); (向上取整)取大于等于x的最接近的double整数

类中各个定义块执行顺序

  • 静态块
    用static声明，JVM加载类的时候执行一次，优先于main函数。

  • 构造块
    比如类定义{}，每一次创建对象时执行，相当于向构造器前面加上构造快的内容

执行顺序: 静态块>构造块>构造方法

继承时执行顺序: 父类静态块>子类静态块>父类构造块>父类构造方法>子类构造块>子类构造方法

运算符规则 - 加法

运行结果： 第4行报错

原因： 表达式类型会自动提升

作加法运算时

  • 所有byte,short,char型的值会被提升为int
  • 如果有一个数字为long型,那么计算结果为long
  • 如果有一个为float时，结果为float
  • 如果有一个为double时，结果为double
