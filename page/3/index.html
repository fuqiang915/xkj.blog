Fragment

片段，官方提供的容器用于展示页面。典型的宿主是FragmentActivity。与Activity的功能类似，但是比Activity更加灵活，可以自由实例化，动态添加、切换、组合等。生命周期随Activity和自己的状态改变。

实际上Fragment不止以Activity为宿主，只要实现了FragmentHostCallback就可以作为Fragment的宿主

原理

Android 3.0(API 11)以后引入，主要是为大屏幕（Pad）提供UI解决方案。

[B] 最好将Fragment设计成可重复使用的模块化Activity组件,一个Fragment可能添加到多个Activity，不要在Fragment直接关联另一个Fragment或Activity，而以接口的方式关联。

参考：官方文档

设计模式--模板方法(类行为型)

概述

模板方法模式是一种比较基本的代码复用技术，父类来控制宏观上几个方法执行步骤，在子类中实现这些方法，这样如果需要修改某个步骤时只去修改子类方法，不需要改变父类逻辑。而且如果有新的实现只要实现一个新的子类就可以了。

场景描述

开发过程中常遇到一种情况：已知一个算法或业务的主要步骤，但每一步的具体实现是未知的，不确定的。

比如去医院看病，一般包含几个步骤：

  1. 挂号
  2. 诊断
  3. 治疗
  4. 离开

挂号和离开是一样的，而其他的诊断和治疗则不尽相同。这时对于各种病症诊断和治疗的实现就是不一样的。

分析

实现上述需求需要考虑几种情况：

  1. 主要步骤大体是固定的，挂号一般是一样的，具体的诊断和治疗实现是不一致的
  2. 但是部分可能有变化，比如诊断结果没有问题的时候不需要治疗，那么治疗这步就不需要了，这就需要根据具体情况确定

模板方法就是抽象出系列固定框架，将一些大体步骤抽象出来，而具体步骤实现延迟到子类中。使得一个算法总体结构不变，可以重新定义算法中某些特定的步骤。(有点类似于注入)

组成

  • 抽象类(AbstractClass): 定义抽象的源于操作，具体的子类将重新定义他们以实现一个算法
  • 具体子类(ConcreteClass): 实现源于操作以完成算法中与特定子相关的步骤

优点

  • 是一种代码复用的基本技术。它们在类库中尤其重要，它们提取了类库中的公共行为。
  • 导致一种反向的控制结构，这种结构有时成为“好莱坞法则”：“Don’t call me,We will call you”。通过父类调用子类的操作（而不是子类操作父类），通过子类的扩展增加新的行为，符合“开闭原则”
  • 符合单一职责原则

缺点

每个不同的实现都要实现一个子类，这会导致类的个数增加，系统更加庞大，抽象，不易理解。

实现

各种方法说明

  • 模板方法: 算法步骤的整体调用
  • 基本方法: 组成算法的每一步具体实现
  • 钩子方法: 用于子类反向控制父类模板方法中调用顺序，逻辑的方法.
    比如父类中通过钩子方法返回值判断某一步骤是否执行，这样子类可以通过返回true or false来控制父类算法的逻辑走向,

设计模式--工厂模式(创建型)

概述

面向对象编程中，最通常的方法是一个new操作时产生一个对象，通常用new操作符来创建。但是一些复杂情况下，new操作符就不够用了。比如一个类的创建需要一系列步骤，可能需要根据参数决定创建哪个子类，生成对象之前需要创建其他的辅助对象，参数需要经过处理等等。这些情况下创建对象就是一个过程了，而不只是一步操作。

问题: 如何能方便的构造对象，而不用关心构造对象的细节和过程呢？

答案就是：工厂模式,工厂模式主要是为创建对象提供过度接口，以便将创建对象的具体过程屏蔽起来，达到灵活性的目的。

工厂模式de历史

  • 还没有工厂时代：假如还没有工业革命，如果一个客户要一款宝马车,一般的做法是客户去创建一款宝马车，然后拿来用。
  • 简单工厂模式：后来出现工业革命。用户不用去创建宝马车。因为客户有一个工厂来帮他创建宝马.想要什么车，这个工厂就可以建。比如想要320i系列车。工厂就创建这个系列的车。即工厂可以创建产品。
  • 工厂方法模式时代：为了满足客户，宝马车系列越来越多，如320i，523i,30li等系列一个工厂无法创建所有的宝马系列。于是由单独分出来多个具体的工厂。每个具体工厂创建一种系列。即具体工厂类只能创建一个具体产品。但是宝马工厂还是个抽象。你需要指定某个具体的工厂才能生产车出来。
  • 抽象工厂模式时代：随着客户的要求越来越高，宝马车必须配置空调。而且这空调必须对应给系列车才能使用。于是这个工厂开始生产宝马车和需要的空调。

最终是客户只要对宝马的销售员说：我要523i空调车，销售员就直接给他523i空调车了。而不用自己去创建523i空调车宝马车.
这就是工厂模式。

分类

工厂模式可以分为三类

  1. 简单工厂模式(Simple Factory)
  2. 工厂方法模式(Factory Method)
  3. 抽象工厂模式(Abstract Factory)

三种工厂的区别

简单工厂模式

工厂方法模式

一个抽象产品类，可以派生出多个具体的产品类
一个抽象工厂类，可以派生出多个具体的工厂类
每个具体工厂类只能创建一个具体产品类的实例

抽象工厂模式

多个抽象产品类，每个抽象产品类可以派生出多个具体产品类。
一个抽象工厂类，可以派生出多个具体工厂类。
每个具体工厂类可以创建多个具体产品类的实例。

多线程知识点总结

线程和进程

进程

进程是运行过程中的程序，进程是操作系统进行资源分配和调度的一个独立单位。

进程的三个特征:

  • 独立: 独立存在的实体，每个进程有自己的内存空间
  • 动态: 程序只是一个静态指令集合，而进程是活动中的指令集合
  • 并发性: 多个进程可在单一处理器上并发执行

并发：指在同一时间点只能有一条指令执行，多个进程指令被快速轮换执行，使宏观上有多个进程同时执行的效果。

并行：同一个时间点，有多条指令在多个处理同时执行。

线程

  • 线程是CPU调度的基本单位
  • 是进程中的实际运作单位，并且在进程中是独立的并发的执行流

进程和线程的区别

  • 线程是进程的组成部分，一个进程可以有多个线程
  • 不同的进程使用独立的内存空间， 而同一个进程的多个线程共享父进程拥有的全部资源。方便了，但也会有同步问题
  • 内存空间和栈内存，每个线程都拥有独立的栈内存用来存储本地手。线程拥有自己的堆栈，自己的程序计数器和自己的局部变量，但不用有系统资源
  • 线程的调度由进程负责，而进程的调度直接由系统来管理和分配资源

  • 就绪

如何中断线程

三种方式中断线程

  • 1,通过代码中设置标记位
  • 2,通过抛出异常停止
  • (不推荐)3,通过stop()方法,stop已经弃用，会造成未知错误

常用方法

sleep(),wait()

sleep()

  • 表示让当前运行的线程休眠一段时间,
  • 时间过后线程自动唤醒,
  • sleep方法是静态的，每次都是针对当前运行中的线程，因为睡眠未运行的线程没有意义。
  • sleep不会释放同步锁

wait()

  • 让线程等待一段时间，需要其他线程唤醒(notify())，
  • wait方法是非静态方法，
  • wait方法会释放同步锁

intercept(),intercepted(),isIntercepted(),stop()

  • intercept(): 将intercept状态设置为true
  • intercepted(): 判断是否已经中断，并将状态值重置。
  • isIntercepted(): 判断是否已经中断

intercept算是Java中提供的中断线程的标准方式（stop已经废弃）。但是调用intercept之后并不会立刻停止线程，而只是改变线程中intercept状态，而线程需要自己判断这个状态值，来处理是否要停止线程（当然自己加状态值来判断是一样的）

stop():弃用

会直接终止线程，有时会抛出ThreadDeadException，会引发未知错误。

为什么不能用stop?

  1. stop会立刻停止线程，这样会引起线程中同步锁强制释放，造成不同步的状态。比如在synchorized方法中间中断返回，会使后面的代码无法执行，可能会有赋值或其他操作，这就导致了不一致的状态。
  2. stop立刻停止线程，使一些线程中需要的回收清理操作无法正确执行。

suspend,resume方法 (弃用)

suspend 暂停线程
resume 继续执行线程

为什么弃用

  • 因为会导致独占资源；比如在synchronized方法中只调用suspend不调用resume会导致同步方法中的资源一直被占用，其他线程阻塞一直无法获取到资源。
  • ?导致不同步；非同步方法中间执行suspend这时其他线程获取值之后再resume会导致不同步

yeild()

放弃当前的CPU执行时间，重新分配CPU资源。有可能会立马再次分配给此线程执行时间。

join()

用来连接两个线程，让他们顺序执行

  • A中调用b.join(),会让A在B之后执行。(A要求加入B)
