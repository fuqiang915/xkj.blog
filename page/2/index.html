面向对象编程

Object Oriented Programing

六大原则：

  • 单一职责: 一个类只有一个改变的动机。一个类只有一个职责。
  • 开闭原则: 对修改关闭，对扩展开放
  • 里氏替换原则: 父类可以由子类替换，而不出现错误
  • 依赖倒置原则: 依赖抽象而不是实现.面向抽象编程，而不是实现
  • 接口隔离原则: 一个类对另一个类的依赖应该建立在最小接口上
  • 迪米特原则: 一个对象应该尽可能独立，对其他类了解越少越好

Monkey日志分析

本文讲解Monkey日志的分析方法，帮助解决常见的Monkey日志中的bug出现位置的查找方法

Monkey

Monkey是Android客户端用于做压力测试的工具，Google官方提供的开源工具。运行Monkey后会根据参数发送随机事件到客户端模拟用户操作来进行压力测试，当出现崩溃时会自动保存相关日志，并发送邮件给开发者。

一个Monkey崩溃日志一般包括三个部分:

  • xxxxxx_x_xxxxxx_bugreport

    这个日志基本包含需要的大部分需要的信息，是重点分析对象，里面包含Android系统的相关信息：

    • 运行中的Activity : Running Application Activities
    • 运行中的Service : Running Services
    • 运行中的Provider : Running Application Providers
    • 系统Log : SYSTEM LOG
    • 事件日志 : EVENT LOG
    • CPU信息: CPU INFO (top -n 1 -d 1 -m 30 -t)
    • 虚拟机内存信息： VIRTUAL MEMORY STATS
    • 虚拟机内存分配情况：VMALLOC INFO
    • 线程信息等等

重点日志一般在系统Log和Activity,里面包含了崩溃信息和Activity的Task信息,大部分错误都可以通过这两种日志找到,至于其他的就按照其他日志按图索骥吧。

搜索 “—– ” 可以定位到每种日志的开头,或者直接搜索上面列出的名称

  • xxxxxx_x_xxxxxx_Monkey_Error

    直接的Android错误堆栈信息,里面一般可以看到出错类的位置，这个可以直接找出错误位置，但是里面指出的错误一般只有一个，但是崩溃可能不止这一个，更多的还是需要在bugreport中寻找。具体信息如下：

  • xxxxxx_x_xxxxxx_Monkey_Standard

    里面是Monkey发送事件的日志，可以看到Monkey 总共事件数和Seed以及所有Monkey发送的随机事件，可以用来了解Monkey所有的测试步骤，对于我们找问题起到的作用不大，如下：

性能分析之-ANR

本篇主要讲解ANR的相关处理方式,包括如何测试，如何调试，如何解决；全面讲解ANR的方方面面.

什么是ANR

ANR是Application Not Response的简写就是应用没有响应。Android应用主线程卡住的时候系统会提示用户是否需要结束掉此App，这是Android系统优化用户体验的一种做法，类似的Windows系统也有类似“程序没有响应”的提示。就是主线程无法及时响应用户输入

什么情况会认为ANR

主线程被阻塞
主线程有耗时操作,比如IO
主线程异常操作比如Thread.sleep,Thread.wait

(Activity)应用在5秒内没有响应用户输入
BroadcastReceiver10秒钟没有响应

常见出现ANR的情况

  • GC时间长导致ANR
    其中一个案例就是，app内页面可以循环打开，不停累加，导致内存占满（实测机器是256MB）此时系统GC时会出现ANR
  • 线程间存在死锁
  • 主线程耗时操作

如何通过日志解决ANR问题

分析ANR的日志一般包含几种： ANR trace日志，系统日志，DropBox日志,Monkey日志(好多ANR都是通过Monkey测出来的)
一般步骤如下

  • 系统日志中找到发生ANR的出现位置，可以通过搜索ANR in关键字可以找到,查看发生在哪个进程中,和相关的trace
  • ANR trace日志中找到anr进程的相关trace详细信息,查找关键信息:Free memory,DALVIK THREADS,”main”
  • Monkey日志可以作为辅助作用，可以看出操作过程，为定位anr位置有帮助

下面详细介绍各个步骤中的详细内容

系统日志(main_log,event_log,sys_log)

查找ANR关键字,初步定位ANR的位置.这个部分难度不大，基本可以找到ANR的类.
这一步需要找到发生ANR的进程名称,相关类名

Android内存泄露问题

最近做的项目内存泄露确实有点严重，目前用leakcanary+MAT结合的方式解决内存泄露问题。下面大致记录一下内存泄露的几种情况。

静态变量static

由静态变量导致的内存泄露还是比较多的，

  • Activity作为全局事件监听器未反注册导致泄露
    项目中有个全局的监听模式实现，用Activity作为监听器，onDestroy时没有移除监听的事件，导致Activity无法释放
  • Activity引用
    Application中持有Activity的静态引用导致泄露(汗~)

单例模式使用不当也会造成泄漏。这里说明一下，单例模式本身并没有什么问题，但是使用时需要注意。单例模式对象一定要在Activity生命周期结束时及时释放对Activity的相关引用，防止Activity无法回收。比较常见的是Context,Listener,View等。

属性动画

属性动画中有一种是无限循环的动画，如果在Activity onDestroy之前没有主动停止，那么会一直播放下去，导致内存泄露

Handler

Hanlder在一些时候应用不当确实会导致短暂的内存泄露。

  • 情况1：匿名Handler会持有外部类的引用，比如定义在Activity中的匿名Handler，在Activity关闭后队列中有没有处理完的消息时，不会释放Handler也就不会释放Activity，造成短暂泄露。
    通常解决方法是，一方面避免使用匿名Handler。而且要在在onDestroy时将Handler的消息队列清空.removeCallbacksAndMessages(null)

Callback,Listener

Callback,Listener 在Activity生命周期结束后及时设为null，或取消注册。

自己定义接口时，当有类似setListener,addListener 一定要对应添加removeListener来释放Listener

第三方SDK：ShareSDK

需要谨慎遵循第三方SDK的生命周期，（能不用就别用吧）

Context ApplicationContext

ApplicationContext代表应用的整个生命周期，我们需要重点注意的是Context的内存泄露问题。 Context 建议用在必须要是Activity的地方，比如资源，View.有一个原则就是能用ApplicationContext就不用Context

Activity启动模式详解

本文介绍Activity的四中启动模式和查看Activity栈的方法

启动模式

standard 常用 多实例

标准模式，是Activity的默认启动模式。此模式下有如下几个特点:

  • 每次启动会新建一个Activity实例并添加到目标Task栈顶

singleTop 常用 可能多实例

此模式与标准模式很类似,可以创建很多实例在同一个Task,除非在栈顶，否则会新建。

特点如下:

  • 每次启动会检查目标Task的栈顶是否是是此Activity的实例
    • 如果是则会复用此Activity并调用它的onNewIntent
    • 如果不在栈顶，会创建一个新的实例到目标Task栈顶

singleTask 单实例

只能存在一个实例在一个Task中,可以创建新Task,只能有一个实例存在

  • 每次会检查默认Task是否存在此Activity
    • 如果存在，会复用此Activity并把其置于栈顶（清除上面其他Activity）,如果它就在栈顶，会复用此实例并调用它的onNewIntent
    • 如果不存在，*会创建一个新的实例放在一个目标Task里 *

注意：这里singleTask官方文档说的是实例会一直处于栈底是不对的，当Activity已经存在时会复用此Activity，而此时它不一定是在栈底的。

singleInstance 单实例

只能存在一个实例，在一个Task中，而且此Task只能存在这一个实例。

目标Task 指taskAffinity指定的Task，默认的taskAffinity是包名

IntentFilter 详解

Intent是Android用于传递信息的包。四大组件的交互都会用到Intent;
IntentFilter是组件定义自己可以处理的内容；用于筛选那些自己可以处理的Intent;

IntentFilter

IntentFilter包含三种数据的定义，`action`,`category`,`data`;IntentFilter通过这三种三个变量来定义，但他们并不一定是必须的；

Action

唯一字符串，表示一个动作，

系统默认提供了一些动作：

打电话: CALL
发送Email: SendEmail

Category

类别字段

类别:
系统默认提供的分类:

DEFAULT : 默认的分类，隐式启动必须

LAUNCHER : 标记的Activity会作为App的启动页面，显示在系统桌面上

Data

携带的数据,Intent中的数据用一段Uri表示格式为 scheme://host/path?arguments=values

七个建议

#一、不要别人点什么，就做什么
“小伙子，千万不要当一个被人点菜的厨师，别人点什么，你就烧什么。不要接受那样一份工作，别人下命令你该干什么，以及怎么干。你要去一个地方，那里的人肯定你对产品的想法，相信你的能力，放手让你去做。”

#二、推销自己
“你工作得很好，代码看上去不错，很少出Bug。但是，问题是别人都没看到这一点。为了让其他人相信你，你必须首先让别人知道你做了什么。你需要推销自己，引起别人的注意。”

#三、学会带领团队
“当你的技术能力过关以后，就要考验你与他人相处的能力了。”

#四、生活才是最重要的
“你要想通，这一切并不重要。有人提交了烂代码，网站下线了，又怎么样？工作并不是你的整个生活。它们不是真正的问题，只是工作上的问题。真正重要的事情都发生在工作以外。我回到家，家里人正在等我，这才重要啊。”

#五、自己找到道路
“以前都是我们告诉你做什么，从现在开始，你必须自己回答这个问题了，我期待你来告诉我，什么事情需要做。”

#六、把自己当成主人
“不要再去开这样的会了。你参加一个会，那是因为你参与了某件事。如果不确定自己为什么要在场，就停下来问。如果这件事不需要你，就离开。不要从头到尾都静静地参加一个会，要把自己当成负责人，大家会相信你的。”

#七、找到水平更高的人
“找到那些比你水平更高、更聪明的人，尽量和他们在一起，吃饭或者喝咖啡，向他们讨教，了解他们拥有的知识。你的职业，甚至你的生活，都会因此变得更好。”
